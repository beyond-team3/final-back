프론트 작업자에게 아래 내용을 기준으로 승인 기능 연동을 반영하고 점검하라.

## 목적
- 견적서 생성 후 별도 승인요청 API를 호출하지 않도록 수정한다.
- 계약서 생성 후 별도 승인요청 API를 호출하지 않도록 수정한다.
- 관리자 승인 목록/대시보드에서 새로 생성한 견적서/계약서가 바로 보이는지 확인한다.
- 수동 승인요청 API는 일반 사용자 흐름에서 제거한다.

## 변경된 백엔드 동작

### 1. 견적서 생성
- 엔드포인트: `POST /api/v1/quotations`
- 기존: 견적서만 생성되고 승인요청은 프론트가 별도 생성해야 했을 수 있음
- 변경 후: 견적서 저장과 동시에 승인요청이 자동 생성됨
- 프론트 액션:
  - 생성 성공 후 `POST /api/v1/approvals`를 추가 호출하지 말 것
  - 성공 응답 수신 후 관리자 승인 목록 화면 또는 대시보드 재조회만 수행할 것

### 2. 계약서 생성
- 엔드포인트: `POST /api/v1/contracts`
- 기존: 계약서만 생성되고 승인요청은 프론트가 별도 생성해야 했을 수 있음
- 변경 후: 계약서 저장과 동시에 승인요청이 자동 생성됨
- 프론트 액션:
  - 생성 성공 후 `POST /api/v1/approvals`를 추가 호출하지 말 것
  - 성공 응답 수신 후 관리자 승인 목록 화면 또는 대시보드 재조회만 수행할 것

### 3. 수동 승인요청 생성 API
- 엔드포인트: `POST /api/v1/approvals`
- 변경 후 정책:
  - 일반 생성 플로우에서는 사용하지 않음
  - ADMIN만 호출 가능
  - SALES_REP/CLIENT에서 호출하면 `403 Forbidden`
- 프론트 액션:
  - 견적/계약 생성 직후 이 API를 호출하는 코드 제거
  - 관리자용 복구/운영 도구가 아니라면 UI에서 노출하지 말 것

## 프론트 점검 시나리오

### 시나리오 A. 견적서 생성
1. SALES_REP로 `POST /api/v1/quotations` 호출
2. 응답 성공 확인
3. 추가 승인요청 API 호출 없이 관리자 화면 재조회
4. 아래 중 하나에서 새 건이 보여야 함
   - 관리자 승인 목록
   - 관리자 대시보드 최근 승인 요청

### 시나리오 B. 계약서 생성
1. SALES_REP로 `POST /api/v1/contracts` 호출
2. 응답 성공 확인
3. 추가 승인요청 API 호출 없이 관리자 화면 재조회
4. 새 계약 승인 요청이 관리자 화면에 보여야 함

### 시나리오 C. 잘못된 구형 흐름 제거 확인
1. SALES_REP가 문서 생성 후 `POST /api/v1/approvals`를 계속 호출하는지 네트워크 탭에서 확인
2. 호출이 남아 있으면 제거
3. 제거 후에도 승인 목록 노출이 정상인지 확인

## 에러/예외 처리
- `POST /api/v1/approvals`
  - SALES_REP/CLIENT 호출 시: `403 Forbidden`
  - 문서 상태가 `WAITING_ADMIN`이 아닌 경우: `400 Bad Request`
  - 이미 진행 중인 승인요청이 있으면: `400 Bad Request`

## 현재 재제출 정책
- 반려 후 수정 저장 기반 재제출 API는 아직 프론트에 새로 열리지 않음
- 따라서 프론트는 “반려 후 수동 승인요청 재생성” 흐름을 만들지 말 것
- 추후 재제출 API가 추가되기 전까지는 기존 문서 생성/승인/조회 흐름만 사용

## 프론트 수정 체크리스트
- 견적 생성 성공 후 `POST /api/v1/approvals` 호출 제거
- 계약 생성 성공 후 `POST /api/v1/approvals` 호출 제거
- 관리자 승인 목록/대시보드 재조회 타이밍 확인
- 수동 승인요청 버튼 또는 훅 사용처 제거 또는 관리자 전용으로 한정
- 403/400 응답을 구형 흐름 잔재 탐지 신호로 활용
