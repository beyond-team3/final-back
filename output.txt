Deal Log / Deal Lifecycle 수정 정리
기준일: 2026-03-13

전제
- INV는 취소 기능을 구현하지 않는다.
- 기존 동작 충돌 방지를 위해 가능한 한 기존 코드 흐름은 유지하고, 필요한 지점에 최소 수정으로 정책을 덧붙인다.
- 수정 대상은 deal log, approval 정리, deal 생성/종결 규칙이다.

확정 정책
1. 새 deal 생성 시점은 새 RFQ 생성 시점이다.
2. ORD, STMT는 삭제 시 deal log에 기록되어야 한다. INV는 취소 기능을 구현하지 않으므로 현 시점 수정 대상에서 제외한다.
3. QUO, CNT, ORD는 삭제 시 진행 중 approval을 정리해야 한다.
4. deal이 닫히는 삭제는 RFQ 삭제뿐이다.
5. deal이 닫히는 비삭제 이벤트는 견적서 만료, 계약 만료 시점이다.

현재 코드 상태 요약
1. RFQ 생성
- 현재는 새 RFQ 생성 시 항상 새 deal을 만들지 않는다.
- QuotationRequestService.resolveOrCreateOpenDeal()가 client 기준 open deal을 재사용한다.
- 관련 파일:
  - src/main/java/com/monsoon/seedflowplus/domain/sales/request/service/QuotationRequestService.java
  - src/main/java/com/monsoon/seedflowplus/domain/deal/core/repository/SalesDealRepository.java

2. RFQ 삭제
- 현재는 상태만 DELETED로 바꾸고 끝난다.
- 삭제 deal log 없음.
- deal.close() 호출 없음.

3. QUO 삭제
- 현재는 상태를 DELETED로 바꾸고 approvalCancellationService.cancelPendingRequest(DealType.QUO, ...)는 수행한다.
- 삭제 deal log 없음.
- deal 종료 없음.

4. CNT 삭제
- 현재는 상태를 DELETED로 바꾸고 approvalCancellationService.cancelPendingRequest(DealType.CNT, ...)는 수행한다.
- 삭제 deal log 없음.
- deal 종료 없음.

5. ORD 삭제
- 실제로는 delete가 아니라 cancel API로 처리한다.
- 상태를 CANCELED로 바꾸고 deal log는 이미 남긴다.
- approval 정리는 없다.
- deal 종료 없음.

6. STMT 삭제
- 실제로는 delete가 아니라 cancel API로 처리한다.
- 상태를 CANCELED로 바꾸고 deal log는 이미 남긴다.
- approval 없음.
- deal 종료 없음.

7. INV 삭제
- 엔티티와 transition policy에는 cancel 개념이 있으나 실제 서비스/컨트롤러 취소 흐름이 없다.
- 정책상 이번 작업에서는 취소 기능을 구현하지 않는다.

8. deal 종료
- SalesDeal.close()는 존재하지만 실제 호출처가 없다.
- 즉 RFQ 삭제, 견적 만료, 계약 만료 어느 쪽도 closedAt을 세팅하지 않는다.

9. approval -> deal log 추적
- submit 로그는 남는다.
- decision 로그도 남는다.
- QUO/CNT는 recordAndSync()를 사용해 deal snapshot 동기화까지 수행한다.
- ORD는 승인 시 confirmOrder()가 별도 recordAndSync()를 수행하고, approval decision 로그는 write()로만 추가 기록한다.
- 현재 approval 추적 자체는 동작하지만 삭제 시 approval cancel 정리와는 연결이 덜 되어 있다.

정책 반영 후 목표 상태
1. RFQ 생성 시 항상 새 deal 생성
- open deal 재사용 제거
- RFQ는 항상 새 deal bootstrap 생성 후 연결

2. RFQ 삭제 시
- RFQ 상태를 DELETED로 변경
- 삭제 deal log 기록
- deal closedAt 설정

3. QUO 삭제 시
- 상태 DELETED 유지
- 진행 중 approval 취소 유지
- 삭제 deal log 추가
- deal은 닫지 않음

4. CNT 삭제 시
- 상태 DELETED 유지
- 진행 중 approval 취소 유지
- 삭제 deal log 추가
- deal은 닫지 않음

5. ORD 삭제 시
- 기존 cancel API/상태 CANCELED 유지
- 진행 중 approval 취소 추가
- 기존 cancel deal log 유지
- deal은 닫지 않음

6. STMT 삭제 시
- 기존 cancel API/상태 CANCELED 유지
- 기존 cancel deal log 유지
- deal은 닫지 않음

7. QUO 만료 시
- 기존 EXPIRED 상태 전환 유지
- 추가로 해당 deal closedAt 설정

8. CNT 만료 시
- 기존 EXPIRED 상태 전환 유지
- 추가로 해당 deal closedAt 설정

수정 원칙
1. 상태 enum은 되도록 그대로 둔다.
- RFQ/QUO/CNT는 DELETED 유지
- ORD/STMT는 CANCELED 유지
- INV는 미수정

2. 기존 API 형태는 되도록 유지한다.
- ORD/STMT는 현재 cancel API를 삭제 정책으로 해석한다.
- 새 delete API를 만들지 않는다.

3. deal log는 문서별 현재 상태명을 사용해 남긴다.
- RFQ/QUO/CNT: toStatus = DELETED
- ORD/STMT: toStatus = CANCELED

4. deal 종료는 별도 helper에서만 수행한다.
- 산발적으로 deal.close()를 직접 호출하지 말고 공통 private helper 또는 전용 domain service로 감싼다.

수정 우선순위
P1. RFQ 생성 시 새 deal 생성으로 변경
P2. RFQ 삭제 시 삭제 deal log + deal close 추가
P3. QUO 삭제 시 삭제 deal log 추가
P4. CNT 삭제 시 삭제 deal log 추가
P5. ORD 취소 시 approval cancel 추가
P6. 견적 만료/계약 만료 시 deal close 추가
P7. 테스트 보강
P8. deal log 관련 문서 정리

2. 수정한 서비스의 단위 테스트
3. approval 테스트
4. deal log 테스트
5. 관련 integration 또는 repository 테스트

최종 주의사항
- INV 취소는 이번 작업 범위에서 제외한다.
- 기존 상태 enum과 API 형태는 되도록 유지한다.
- DealPipelineFacade, DocStatusTransitionPolicy는 영향 범위가 넓으므로 마지막까지 수정 최소화 전략을 유지한다.
- deal.close()는 RFQ 삭제, 견적 만료, 계약 만료 세 경우에만 호출한다.
